Writing scenarios

We used extensively Cucumber, and what they call Behavior-Driven-Development (BDD). I guess that what we are trying to do, is to achieve the goals that they are evangelizing, but with less work and more lever on various tools.

// In BDD, a scenario is a sequence of such statements:
GIVEN some existing context/state, WHEN I perform action ..., THEN we should have the result/state ...

// Rule of thumb to decide between GIVEN and WHEN:
// GIVEN -> the code for this step will prepare some data (via mocking
// or via copy into BD), in order to recreate a state that happens in practice
// WHEN -> the code for this step will simulate an user action
// NOTE: In many cases we have only WHEN + THEN; or sometimes also GIVEN + THEN
GIVEN a paid invoice, THEN a green sticker labeled "PAID" appears in the top right corner
WHEN click on UP, THEN the Todo slides up

// We may use AND
WHEN click on Delete, AND click on "yes, I confirm", THEN ...
// Purists will say that tests should be super small. But in practice sometimes it's 
// easier to understand "the big picture" with 1 longer test (w/o exagerating), 
// compared to having 2 or more smaller ones. 
WHEN click on Delete, THEN alert appears, AND WHEN click on "yes, I confirm", THEN ...

Shorthands​

// Shorthand for something like:
// GIVEN a Todo w/ fields: name = "Buy bread", priority = "Urgent", index = "3",
// WHEN the screen renders,
// THEN the label for name = "Buy bread", priority = "Urgent" and colored in red, ...

// We notice that it's awkward to use GIVEN, WHEN, THEN in this case. But this is a feature
// which needs to be tested as well. Hence we introduce such new key words/shorthands.

// It's subjective for what (and if at all!) we use RENDERER.
// A RENDERER is usually linked to a model element (e.g. "bigger", like an entity/DB table, or 
// "smaller", like something used client side only). 
RENDERER for a Todo

// Similar to RENDERER. But bigger. In fact so big that it takes an entire screen / route :)
// Usually such scenarios have a drawing/wire frame attached. We shouldn't try to write in words
// what is on the drawing. They will normally appear any way in the Tests Are Demo test.
SCREEN for a dashboard of Todos

// Similar to RENDERER, but each field can also be edited
FORM for a Todo

// Useful when our deliverable is a component and the user is a dev. The associated Tests Are Demo 
// test will show the various behaviors of this property. Often we talk about one or two WHEN/THENs.
// But there are cases when there are more, and in various locations. E.g. WHEN render ..., WHEN create new ...,
// GIVEN user has permission ..., WHEN ..., etc.
PROPERTY showWeekends
Convention for uniformity and readability

Please respect the casing (e.g. WHEN and not When or when), and the comma (,) before the key words (e.g. ... something, THEN something else).

Difference to Cucumber​

We still have hundreds of Cucumber files. Some examples:

Feature: 
  Scenario: editContact(): Navigation to form
    Given the renderer is given a Contact with id = 2
    When the user clicks the edit contact button
    Then navigation is attempted to "/properties_form/{id}" with id = "2"
Feature: 
  Scenario: onReveal(): Get all contacts
    Given the DB contains the contacts
      | name  | age | married | city      |
      | john  |  30 | true    | Bucuresti |
      | mary  |  16 | false   | Brasov    |
      | johan |  20 | false   | Bucuresti |
    And a mocked, mirroring ContactSorterComponent
    And a mocked, mirroring SearcherComponent 
    When ContactListPage is revealed
    Then the list has ContactRenderers displaying the following names
      | john | mary | johan |
    And the telescopic invocations were OK for ContactSorter and Searcher components

The disadvantage is that there is a lot of text. And often it's copied/pasted between scenarios/steps w/ small modifications. The attention span of the reader is consumed after a couple of scenarios. And if there are errors hidden in the text, they are super easy to miss.

Feature Book offers a dual view: overview + detail. It splits a scenario in:

  • the overview = the scenario title;
  • the detail = the Tests Are Demo test.

In other words:

Feature BookCucumber
Size of a scenarioOne line. We recommend not to overcome (maybe w/ exceptions) the width of a screenSeveral / many lines
How detailed?We are not precise. We express in a general way; e.g. "WHEN we create a Todo"We need to be specific, e.g. "WHEN we create a Todo w/ name = Buy bread, priority = Urgent, ..."
How exhaustive?Our text doesn't cover all the detailsEach scenario specifies/tests all the details
