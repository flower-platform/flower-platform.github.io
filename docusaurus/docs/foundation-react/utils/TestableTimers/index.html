TestableTimers

The technique to test code that uses timers is the following:

  • Before the test, we replace the timers w/ a mocks.
  • During the test, from the test code, we trigger ourselves the timer by invoking the mock. I.e. we control time.
  • E.g. if a code adds a timer for 5 seconds, our test won't wait those 5 seconds. Our test code will say "the 5 seconds have passed; please continue".

TestableTimers.ts helps us in applying this technique. This class implements the same interface as FakeTimers.createClock() from Sinon.JS / Fake timers lib. Sinon.JS is a popular library for dealing w/ mocks in JS.

1

4
In the test, "control the time". I.e. after a function that is using a timer is invoked, fire the timer "manually".

note

During tests, ClientErrorLogger.timers: TestableTimers and clock: BrowserClock are the same instance. But in the test we manipulate it using clock, because the BrowserClock type is richer, exposing methods such as runAllSync().

note

The @sinonjs/fake-timers have other interesting functions. Depending on your case, they might come in handy. The library is even capable of mocking global functions such as setTimeout(). However, this is not a good practice. Maybe you'll interfere w/ other timers, that you don't want to mock. A good example is the TestsAreDemo library. Mocking the global timer functions breaks things in the library.
