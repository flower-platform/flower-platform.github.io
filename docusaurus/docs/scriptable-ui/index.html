scriptable-ui

What? Like macros in Excelâ€‹

Do you remember macros in Excel?
  • You press a Record button, you do some operations in Excel (select, add, modify cells, etc.) and a macro is recorded.
  • You press Play, and the macro runs, doing exactly what you did previously (and has been recorded).
  • The macro is actually generated VBA code. If you know a bit of programming you can adjust it a bit (or even a lot), adding a more dynamic behavior.

By using the scriptable-ui lib, you'll have similar features for your app.

Why?â€‹

The primary need that drove us to develop this lib is to use it in combination w/ TestsAreDemo. Our ambition regarding tests:

  • we want to spend the minimum of time to write them;
  • and we want them to have a maximum of coverage;
  • and then even more, we want to reuse the tests as demo + training purposes.

A test has basically part 1 = simulate an action, and part 2 = verify that the results are the expected ones. scriptable-ui helps for part 1.

And it helps a lot actually. All what the developer needs to do is: click, click, etc, and then copy/paste the generated script in the test file.

...ScriptableUi (a.k.a. the "man in the middle") is actually a proxy that has 3 modes:

  • 1/ By default, it does nothing special; i.e. we are talking about intermediary functions that just delegate
  • 2/ During Record mode, it intercepts the call and invites the user to record the action in a script:

  • 3/ During Play mode, the actions don't come any more from the user. But from the recorded script.

A family of functional components (main + children)â€‹

In Calculator4_nativeScriptableUiWithSubComponent.tsx we illustrated how to use scriptable-ui with more components. We see the interest of using a namespace to group ScriptableUiCalculator4.Main and ScriptableUiCalculator4.OperatorButton.

We already mentioned that these ScriptableUi... classes are a bit like an "API" of the UI. We would have liked to have one "API" per component. But this is not easy for complex components (that have a lot of subcomponents). Using a namespace is a good compromise. Such classes should always be prefixed w/ ScriptableUi.... This way, when using content assist, it shows good proposals.

Some conception detailsâ€‹

About idsâ€‹

When the script executes, for each instruction it needs to identify the ScriptableUi instance corresponding to a component. If we are talking about a component that is a singleton (e.g. a page), then there is no big deal. However, if we are talking about components that have multiple instances, then it's mandatory to have unique ids, so that they can be identified from the script. Uniqueness is needed only under the "hat" of the parent component (that has ScriptableUi). This is because for the final id, ids of parent components are prepended, forming a bigger / qualified id.

About HWâ€‹

... which is an alias for ScriptableUiHighlightWrapper. We need to wrap the component that generated the initial event w/ HW because we want a nice visual feedback, i.e. a spotlight around:

  • the component that dispatched the event (in record mode),
  • respectively the component that will be "simulated" via script (in play mode / step by step).
