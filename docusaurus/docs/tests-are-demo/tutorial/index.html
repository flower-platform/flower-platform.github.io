Define the scenariosâ€‹

A very good intellectual exercise is to define the scenarios that we want to implement, before doing the actual implementation. This adds structure to our thinking, and defines clearly our development road map.

At this stage, the scenarios are empty functions. A scenario == a test == a feature.

tip

We noticed in practice that this technique: "think about the scenarios before writing the code" has as side effect the creation of a mental structure of the code. This may also include quite soon, the identification of the sub components to use. E.g. check / uncheck a Todo => checkboxes => we need a renderer for each Todo => let's name it TodoItem.

Adding the @Scenario annotation, makes the function being a "runnable test". It's the equivalent of:

    it("WHEN click ...", () => { ... } )

The title of the scenario (i.e. the ... from @Scenario(...)) uses the BDD "syntax". The function name is kind of abbreviation (but very short) of the scenario title.

export class TodosTestsAreDemo {

    ...

    @Scenario("WHEN click on DELETE, THEN confirm, AND WHEN 'yes', THEN the Todo is removed")
    async whenDelete() {
        // DRAFT: click on del, verify alert open, click no, verify record still there, 
        // click again on del, click yes, verify the record is not there
    }

    ...
}

For a lot of scenarios, the title (almost) fully describes the feature. But there are scenarios that contain small (and important) details. In this case, we write them down in a comment/draft, to avoid forgetting them. Once we do the implementation + test, we'll delete this comment/draft, because the small (and important) details will be visible in the test.

Let's look at the example above. We have a confirmation/alert of type "Do you really want to delete?". It's obvious that if we click on no, nothing should happen, and the alert should close. So we don't want to pollute the scenario title with such a detail. But testing (and implementing) the detail is important.

tip

One would be amazed how often we saw regressions related to (more or less) similar things. And believe us, an user will get really mad if e.g. he/she spends say 15 minutes in an editor, and then hits by mistake F5 or Back, and the dev forgot to implement a "leave page" handler + alert.

Use @Comment(...) for comments that you want to see in the UI. For example we used them here to kind of group the scenarios per component. Don't forget that JS supports multi line comments w/ backticks: `...`.

Implement the first scenarioâ€‹

BDD styleâ€‹

We like BDD (behavior driven development) which is a type of TDD (test driven development) or TFD (test first development). BDD and the RED/GREEN practice (cf. Cucumber doc) is quite good for developing server code. E.g.: we write all the tests first (skeleton + implementation), we run test #1 => it fails (it's RED), we implement, we run => test passes (it's GREEN). And we iterate w/ test #2, #3 and so on. When all is GREEN => the job is done.

However we find that:

  • This approach is not reasonable for UI development.
  • A good compromise is: the fact that we define the scenarios (only skeleton; no impl) before the code, and then for each scenario => we implement + test.

Back to our tutorial: so the scenarios were defined, and now we take them one by one and we implement + test them, cf. below.

We'll focus on the "delete" scenario. The implementation is straight forward: a "Delete" button on each line/todo. On click => an alert asking for confirmation. On confirm => the corresponding element is removed from the list.

Flow: select, perform action, select, verify resultâ€‹

A test is basically a repetition of these operations:

  • Step 1, select a DOM element. We use React Testing Library.
  • Step 2, perform an action (on it). We use user-event (related to React Testing Library).
  • Step 3, select another DOM element.
  • Step 4, verify something (on it).

We use Mocha as the underlying framework. At the time of writing it's the only runner that has an implementation for the browser, which we need for the demo part.

For the demo part, Tests Are Demo intercepts/wraps the 3 categories of functions (select DOM element, perform action, verify). When selecting a DOM element, Tests Are Demo captures/remembers it. When performing an action or verification, Tests Are Demo will show a spotlight and a popup (focused around the captured element) and will wait for the user clicking on "Next" step.

Add data-testid (if not existing)â€‹

There are multiple ways to select a DOM element. We recommend adding data-testids to components, and use them as selection criteria.

React Testing Library currently recommends avoiding to (over)use the data-testid attribute for selecting DOM elements. Our experience has taught us the contrary.

Todos.tsx
export const todosTestids = createTestids("Todos", {
    ...
    todoItem: "", todoItemCheckbox: "", todoItemCheckboxLabel: "",
    ...
    // TodoItem
    up: "", down: "", edit: "", delete: "",
    deleteYes: "", deleteNo: "",
    ...
});
Todos.tsx
const TodoItem = ({ todo, ...props }: TodoItemProps) => {
    ...
    return (<Segment data-testid={todosTestids.todoItem + "_" + props.index} onDoubleClick={props.onEdit} ...>
        ...
        <div>
            ...
            <Button data-testid={todosTestids.delete} onClick={() => setModalOpen(true)} basic icon="delete" color="red" content="Delete" />
}

At runtime, todosTestids.delete == "Todos_delete". We use todosTestids.delete, to have type checking and autocomplete. This way:

  • we cannot misspell (e.g. Todos_delete vs Todos_delte); misspelling would be eventually discovered anyway (because the test wouldn't pass), but w/ additional cost;
  • we have traceability from the implementation <-> test (via CTRL + click). This is a helpful feature ðŸ˜‰.
For uniformity

Always keep data-testid as the first property of a component. And BTW, did you notice that, for todosTestids, we grouped (+ commented) the testids by component? And arranged the fields in sort of a space saving manner?

Now that the data-testids are there, we need to write code that uses them. Tests Are Demo UI has a handy helper feature: the Inspect testid button:

async whenDelete() {

note

In this case, we used the pasted the code snippet "as is". No modifications/adaptations were needed, which is rather rare. Normally a bit of adaptation is needed to the pasted snippet. So please consider this copy/paste workflow as an aid, not a bullet proof mechanism!

info

1/ in Todos.tsx we have todosTestids. 2/ here, in TodosTestsAreDemo.tsx we have testids. Hence we rename the import:

TestsAreDemo.tsx
import { ..., todosTestids as testids } from "./Todos";

1/ is so, so that we don't pollute the content assist (e.g. type "testids" => tons of proposals). 2/ is so, for easier snippet genaration.

Run. The "demo" mode works by intercepting function callsâ€‹

After running the test (w/ the option "Step by step" = ON [default]), we have this result:

Let's explain what is happening. We compare Tests Are Demo code with "plain" React Testing Library code. For the selection of the DOM element:

// 1/ Tests Are Demo code
let item = tad.screenCapturing.getByTestId(testids.todoItem + "_0");
// 2/ normal React Testing Library code
let item = screen.getByTestId(testids.todoItem + "_0");

tad.screenCapturing().getByTestId() (from 1/) is a wrapper around screen.getByTestId() (from 2/). 1/ calls 2/. All the wrapped functions behave identically, except that they capture and remember the DOM element that was selected. Hence the suffix "Capturing". We grouped all TAD functions into tad, to make our lib "content assist" friendly. I.e. type tad. and see what are the available functions, w/o needing to go to the doc.

To perform an action, tad.userEventWaitable() (from 3/) is also a wrapper:

// 3/ Tests Are Demo code
await tad.userEventWaitable.click(tad.withinCapturing(item).getByTestId(testids.delete));
// 4/ normal React Testing Library code
userEvent.click(within(item).getByTestId(testids.delete));

3/ captures the action that will be performed (the function name, e.g. click), it displays a "spotlight" around the last captured element (e.g. the "Delete" button), it shows a popup saying "Will perform click", and then waits. When the user does something, e.g. click on "Next", then 4/ is called, and the test continues. Hence the suffix "Waitable" and the function being async. We chose this suffix so that people can remember that they also need to use await.

Select DOM element + verify (again, via snippet)â€‹

We apply the same process to obtain a snippet for verification:

TodosTestsAreDemo.tsx
@Scenario("WHEN click on DELETE, THEN confirm, AND WHEN 'yes', THEN the Todo is removed")
async whenDelete() {
    let item = tad.screenCapturing.getByTestId(testids.todoItem + "_0");
    await tad.userEventWaitable.click(tad.withinCapturing(item).getByTestId(testids.delete));

    await tad.userEventWaitable.click(tad.screenCapturing.getByTestId(testids.deleteNo));
    tad.cc("The Todo wasn't deleted");
    await tad.assertWaitable.equal(tad.withinCapturing(item).getByTestId(testids.todoItemCheckboxLabel).textContent, someTodos[0].text);

We have the result:

We added a new "perform action" line, simulating clicking on "No" (in the alert that's opened after clicking on "Delete"). And then a "verification" line. Here we have the same interception mechanism, but applied this time to assert from Chai Assertion Library.

If the first lines were quite readable, this particular verification line is not straight forward. Meaning that just by reading it, it's not clear what we want to achieve actually. Hence the need for a comment, i.e. tad.cc() which is an alias for tad.currentComment(). We use comments this way (instead of // this way), so that they can be displayed in the UI.

Repeat until scenario is finishedâ€‹

To finish the test/scenario, we repeat the steps above. Let's recap them:

  • add data-testid (if needed),
  • copy/paste/adapt code for performing actions,
  • copy/paste/adapt code for verification.
tip

We recommend keeping "Auto run after page load" = true (default). Probably you are running Tests Are Demo inside a toolchain (such as Vite or Create React App) that has auto-refresh or hot module reload (HMR) enabled. Hence when you save, the browser will refresh and the test will run again. This is convenient.

Polish the demo for the end userâ€‹

The "slide show"/visual test steps are practically a demo of our component or app. They are meaningful for a dev that looks at the demo. And they are also (almost) meaningful for an end user. However, when presenting the demo to and user, a bit of additional polishing may be needed.

TodosTest.tsx
@Scenario("WHEN click on DELETE, THEN confirm, AND WHEN 'yes', THEN the Todo is removed")
async whenDelete() {
    tad.cc("A Todo may be deleted");

    tad.demoForEndUserHide();
    let item = tad.screenCapturing.getByTestId(testids.todoItem + "_0");
    await tad.userEventWaitable.click(tad.withinCapturing(item).getByTestId(testids.delete));

    await tad.userEventWaitable.click(tad.screenCapturing.getByTestId(testids.deleteNo));
    tad.cc("The Todo wasn't deleted");
    await tad.assertWaitable.equal(tad.withinCapturing(item).getByTestId(testids.todoItemCheckboxLabel).textContent, someTodos[0].text);
    tad.demoForEndUserShow();

    await tad.userEventWaitable.click(tad.withinCapturing(item).getByTestId(testids.delete));
    await tad.userEventWaitable.click(tad.screenCapturing.getByTestId(testids.deleteYes));
    tad.cc("The Todo was deleted. So the second element (index 1) became element the first element (index 0)");
    item = tad.screenCapturing.getByTestId(testids.todoItem + "_0");
    await tad.assertWaitable.include(tad.withinCapturing(item).getByTestId(testids.todoItemCheckboxLabel).textContent, someTodos[1].text); // include instead of equals, because we appended something in a step above
}

Some ways to polish:

  • by adding additional comments. Such as providing some context or intro at the beginning of the scenario, or adding text to mark the transition from the previous scenario.
  • by hiding steps which would bore the end user. E.g. here we don't want to show that we test clicking on the "No" button of the "Do you want to delete?" dialog.
  • by rephrasing a comment using a language suitable for the end user. E.g. here we used both "first element" (in "user language") and "index 0" (in dev language).

Tips for implementing the remaining scenariosâ€‹

temp

We already explained that the UI doesn't allow selecting a "current" file. Hence during dev, we need to comment the other files.

Regarding individual scenarios (tests), we have the @Only annotation. If we annotate a certain function with it, in principle, only that scenario will run. This is inspired from Mocha's it.only(...).

Linking scenariosâ€‹

Let's look at this:

TodosTestsAreDemo.tsx
@Scenario("WHEN click on Add, THEN the form opens")
@ScenarioOptions({ linkWithNextScenario: true })
async whenAddThenForm() {
    tad.cc("Let's add a new Todo");
    await tad.userEventWaitable.click(tad.screenCapturing.getByTestId(testids.add));
}

@Scenario("FORM for a Todo; test #1")
async formTodo1() {
    tad.cc("Please observe that this text field is already focused. So you can directly start typing, w/o needing to click here first");
    const textbox = tad.withinCapturing(tad.screenCapturing.getByTestId(testids.inputText)).getByRole('textbox');
    ...

Linking scenarios helps us write less code:

  • whenAddThenForm() doesn't have a verification part. The verification is practically done in the next scenario. If the form wouldn't open, the next scenario would fail while finding testids.inputText.
  • formTodo1() doesn't have a part that opens the form. Because this is done by the previous scenario.

The two scenarios are linked/dependent, and they "help" each other. And we wrote less code, compared to the case where the two scenarios are independent. Less code to achieve the same result is always good, but in particular in the "test business", where test clutter is a constant threat.

When one scenario is ran individually (e.g. via @Only or via UI), then the other is ran as well. Chaining more than 2 scenarios is also possible.
