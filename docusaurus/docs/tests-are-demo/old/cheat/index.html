"Cheating", a reasonable compromise for hard to test cases

Overview​

In some cases, it may be "exponentially" hard:

  • to simulate that we perform an action
  • and/or obtain corresponding values to verify.

The slider in the Todo form may be such a case:

If we want to position the slider to the value 50, it's difficult to dictate the amount of pixels to drag. Taking into account the screen size, dpi, maybe future evolutions of the screen, etc.

We propose a compromise that we find reasonable. From within the test code:

  • "cheat" by skipping the actual user interaction simulation;
  • and call instead the function, that the user interaction would call.

In the component​

The component needs to "collaborate" with the test:

Todos.tsx
export class Slider extends React.Component<{ value?: number, onChange: (newValue: number) => void }> {

    onChange(newValue: string) {
        this.props.onChange(Number(newValue));
    }

    render() {
        return <>
            <TestsAreDemoCheat objectToPublish={this} />
            <input type="range" min={0} max={100} step={10} value={this.props.value || 0} onChange={v => this.onChange(v.target.value)} />
        </>
    }
}

Here collaboration means:

  • the use of <TestsAreDemoCheat>. A ref to the object will placed in the DOM, so that we can find it during the tests.
  • the fact that we have onChange() as separate (thus callable) function. Without Tests Are Demo in mind, we could have written the code without it.

From test: "perform action" / "waitable" mode​

Here, we use the "cheat" in the "waitable" / "perform action" mode:

@Scenario("FORM for a Todo; test #2")
async formTodo2() {
    ...
    // it's difficult to simulate a precise mouse drag; hence using a cheat
    await tad.getObjectViaCheatWaitable(Slider, "slide").onChange("50");

This is how it looks in the "step by step" mode:

This solution works out of the box for class components. In many cases, the functions that we'd want to call from the test: they exist already. However, there may be cases where we need to create a new function, specially for the test. For code organization purposes, we recommend to prefix it e.g. tadAddOption(...), and clearly specify in its doc that its meant to be used for the test.

From test: "get value" mode​

Here, we use the "cheat" in "get value" mode. We obtain the value from the "internals" of the component, and then compare it w/ what was expected.

TodosTestsAreDemo.tsx
@Scenario("FORM for a Todo; test #2")
async formTodo2() {
    ...
    tad.demoForEndUserHide();
    // "Percentage done" is currently displayed as a number; so it's easy to verify it the "normal way".
    // However, 1/ we want to demo the use of "cheat" in a functional component. 
    // And 2/ it could use a fancier renderer. E.g. as SVG progress bar or pie chart.
    // Hence using a cheat.
    await tad.assertWaitable.equal(tad.getObjectViaCheat(TodoItemCheat, item).getTodo().percentageDone, 50);
    await tad.assertWaitable.include(item.className, colorLow);

Using <TestsAreDemoCheat> in a functional component​

If we look carefully at the two tad.getObjectViaCheat*() from above, we'll notice that in the first case we accessed the actual instance of the class component. And in the second case we used a class named TodoItemCheat. Creating an intermediate class: is the recipe we propose for the case of a functional component. Here, the goal of this class, is to give access to internal data from within the component.

Todos.tsx

// Because TodoItem is not a class component, we cannot expose it via "cheat". We need an actual
// class for this. 
export class TodoItemCheat {
    
    // We may consider this an "abstract" function
    getTodo!: () => Todo;
}

const TodoItem = ({ todo, ...props }: TodoItemProps) => {
        ...
        <TestsAreDemoCheat objectToPublish={new class extends TodoItemCheat {
            getTodo = () => todo;
        }} />
        <div data-testid={todosTestids.todoItemCheckbox} className="TodoItem_left">
        ...
