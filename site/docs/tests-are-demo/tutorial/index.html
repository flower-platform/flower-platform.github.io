Tutorial

Intro​

We present the recommended workflow for using TestsAreDemo.

For this tutorial, we suppose that we are developing Todos.tsx, a component that is managing things to do. It looks like this:

While developing this component, we used TestsAreDemo in TodosTest.tsx. This tutorial shows the steps we followed to create this file.

This is the result/the "demo" part (in TestsAreDemo):

Create the files and start an empty test​

We created the 2 files below (click on tabs):

  • TodosMeta.ts
  • TodosTest.tsx

This file will eventually look like this: TodosMeta.ts

import { render } from "@famiprog-foundation/tests-are-demo";
import { tad } from "@famiprog-foundation/tests-are-demo";
import { TodosMeta } from "./TodosMeta";

const { testids, scenarios } = TodosMeta;

describe("Todos", () => {

    before(async () => {
        render(<Todos />);
        await tad.waitForCommunicationFinished();
    })

    it("empty", async () => {
        console.log("Hello world");
    })
}

Define the scenarios​

A very good intellectual exercise is to define the scenarios that we want to implement, before doing the actual implementation. This adds structure to our thinking, and defines clearly our development road map.

TodosMeta.ts
export const TodosMeta = createMeta("Todos", {
    scenarios: {
        // Todos (func comp)
        rendererTodos: "RENDERER for a list of Todos",
        rendererTodo: "RENDERER for a Todo",
        whenAddThenForm: "WHEN click on Add, THEN the form opens",

        // TodoItem (func comp)
        whenCheckbox: "WHEN click on checkbox or label, THEN toggle 'done'",
        ...
        // DRAFT: click on del, verify alert open, click no, verify record still there, 
        // click again on del, click yes, verify the record is not there
        whenDelete: "WHEN click on DELETE, THEN confirm, AND WHEN 'yes', THEN the Todo is removed", 
        ...

        // TodoForm (func comp)
        formTodo: "FORM for a Todo"

The scenario is a string (we will discuss the format below). And it has a field name, which should be rather short (although when we'll reference it, the "auto complete" of the IDE will do most of the typing).

export const TodosMeta = createMeta("Todos", {
    scenarios: {
        ...
        // click on del, verify alert open, click no, verify record still there, click again on del,
        // click yes, verify the record is not there
        whenDelete: "WHEN click on DELETE, THEN confirm, AND WHEN 'yes', THEN the Todo is removed", 
        ...

For a lot of scenarios, the title (almost) fully describes the feature. But there are scenarios that contain small (and important) details. In this case, we write them down in a comment/draft, to avoid forgetting them. Once we do the implementation + test, we'll delete this comment/draft, because the small (and important) details will be visible in the test.

Let's look at the example below. We have a confirmation/alert of type "Do you really want to delete?". It's obvious that if we click on no, nothing should happen, and the alert should close. So we don't want to pollute the scenario title with such a detail. But testing (and implementing) the detail is important.

tip

One would be amazed how often we saw regressions related to (more or less) similar things. And believe us, an user will get really mad if e.g. he/she spends say 15 minutes in an editor, and then hits by mistake F5 or Back, and the dev forgot to implement a "leave page" handler + alert.

        // Todos (func comp)
        rendererTodos: "RENDERER for a list of Todos",
        ...

        // TodoItem (func comp)
        whenCheckbox: "WHEN click on checkbox or label, THEN toggle 'done'",
        ...

        // TodoForm (func comp)
        formTodo: "FORM for a Todo"
        ...

Often the file holds the main component + some additional smaller components (functions or classes). It's nice to group the scenarios by component name.

tip

We noticed in practice that this "think about the scenarios before writing the code" has as side effect the creation of a mental structure of the code. This may also include quite soon, the identification of the sub components to use. E.g. check / uncheck a Todo => checkboxes => we need a renderer for each Todo => let's name it TodoItem.

For the title of the scenario, we use keywords inspired from Cucumber, a BDD (behavior driven development) technology. E.g.:

// default format
GIVEN something, WHEN something, THEN something

// Rule of thumb to decide between GIVEN and WHEN:
// GIVEN -> the code for this step will prepare some data (via mocking
// or via copy into BD), in order to recreate a state that happens in practice
// WHEN -> the code for this step will simulate an user action
// NOTE: In many cases we have only WHEN + THEN; or sometimes also GIVEN + THEN
GIVEN a paid invoice, THEN a green sticker labeled "PAID" appears in the top right corner
WHEN click on UP, THEN the Todo slides up

// we may use AND
WHEN click on Delete, AND click on "yes, I confirm", THEN ...
// purists will say that tests should be super small. But in practice sometimes it's easier to have
// 1 longer test (w/o exagerating), than to have 2 smaller ones
WHEN click on Delete, THEN alert appears, AND WHEN click on "yes, I confirm", THEN ...

// these are sort of shortcuts. It's obvious what they mean. 
// Renderers and forms/editors need to be tested, but it would be incredibly
// awkward to say (and repeat for each field) things such as: 
// GIVEN a Todo w/ text = "Buy bread", THEN the label shows "Buy bread". 
RENDERER for a Todo
FORM for a Todo
Convention for uniformity and readability

Please respect the casing (e.g. WHEN and not When or when), and the comma (,) before the key words (e.g. ... something, THEN something else).

Implement scenario #1​

We like BDD (behavior driven development) which is a type of TDD (test driven development) or TFD (test first development). BDD and the RED/GREEN practice (cf. Cucumber doc) is quite good for developing server code. E.g.: we define the structure, we write all the tests first, we run test #1 => it fails (it's RED), we implement, we run => test passes (it's GREEN). And we iterate w/ test #2, #3 and so on. When all is GREEN => the job is done.

However we find that:

  • This approach is not reasonable for UI development.
  • It's a good compromise: the fact that we define the scenarios before the code, and then for each scenario => we implement + test.

So the scenarios were defined, and now we take them one by one and we implement + test them, cf. below.

We'll focus on the "delete" scenario. The implementation is straight forward: a "Delete" button on each line. On click => an alert asking for confirmation. On confirm => the corresponding element is removed from the list.

Step 1: Add data-testid (if not existing)​

A test is basically a repetition of these operations:

  • select a DOM element. We use React Testing Library.
  • perform an action (on it). We use user-event (related to React Testing Library).
  • verify something (on it or on another selected DOM element). We use Mocha.

For the demo part, TestsAreDemo intercepts/wraps these 3 categories of functions. When selecting a DOM element, TestsAreDemo captures/remembers it. When performing an action or verification, TestsAreDemo will show a spotlight and a popup (focused around the captured element) and will wait for the user clicking on "Next" step.

React Testing Library currently recommends avoiding to (over)use the data-testid attribute for selecting DOM elements. Our experience has taught us the contrary. Hence we begin by adding a data-testid in our code.

TodosMeta.ts
export const TodosMeta = createMeta("Todos", {
    ...
    testids: {
        todoItem: "",
        ...
        // TodoItem
        delete: "", deleteYes: "", deleteNo: "",
        ...
    }
});
Todos.tsx
const TodoItem = ({ todo, ...props }: TodoItemProps) => {
  ...
  return (<Segment data-testid={testids.todoItem + "_" + props.index} onDoubleClick={props.onEdit} ...>
    ...
    <div>
        ...
        <Button data-testid={testids.delete} onClick={() => setModalOpen(true)} basic icon="delete" color="red" content="Delete" />

testids.delete == "Todos_delete". We use TodosMeta.testids, to have type checking and autocomplete. This way:

  • we cannot misspell (e.g. Todos_delete vs Todos_delte); misspelling would be eventually discovered anyway (because the test wouldn't pass), but w/ additional cost
  • we have traceability from the implementation <-> test (via CTRL + click)
For uniformity

Always keep data-testid as the first property of a component. And BTW, did you notice, in TodosMeta.ts that we grouped (+ comment) the testids by component? And arranged the fields in sort of a space saving manner?

Step 2b: Paste (and adapt) code snippet​

Paste the snippet within the test.

TodosTest.tsx
it(scenarios.whenDelete, async () => {
    let item = tad.screenCapturing.getByTestId(testids.todoItem + "_0");
    await tad.userEventWaitable.click(tad.withinCapturing(item).getByTestId(testids.delete));
}

After running the test (w/ the option "Step by step" = ON [default]), we have this result:

Let's explain what is happening, and start by comparing "plain" React Testing Library code with TestsAreDemo code:

// normal React Testing Library code
let item = screen.getByTestId(testids.todoItem + "_0");
// TestsAreDemo code
let item = tad.screenCapturing.getByTestId(testids.todoItem + "_0");

tad.screenCapturing() is a wrapper around screen. All the functions behave identically, except that they capture and remember the DOM element that was selected. Hence the suffix "Capturing". We grouped all TAD functions into tad, to make our lib "content assist" friendly. I.e. type tad. and see what are the available functions, w/o needing to go to the doc.

// normal React Testing Library code
userEvent.click(within(item).getByTestId(testids.delete));
// TestsAreDemo code
await tad.userEventWaitable.click(tad.withinCapturing(item).getByTestId(testids.delete));

tad.userEventWaitable() is also a wrapper. It captures the action that will be performed (the function name, e.g. click), it displays a "spotlight" around the last captured element (e.g. the "Delete" button), it shows a popup saying "Will perform click", and then waits (for the user to do something, e.g. click on "Next"). Hence the suffix "Waitable" and the function being async. We chose this suffix so that people can remember that they also need to use await.

note

In this case, we used the pasted code snippet "as-is". No modifications/adaptations were needed, which is rather rare. Normally a bit of adaptation is needed to the pasted snippet. So please consider this copy/paste workflow as an aid, not a bullet proof mechanism!

Regarding the name of the test, i.e. scenarios.whenDelete. Again, we leverage typing. E.g. a scenario may be tested in several tests. Hence we achieve traceability (via CTRL + click). We also have the case where a test covers several scenarios, e.g.:

Step 3: Copy/paste code snippet (verify)​

We apply the same process to obtain a snippet for verification:

TodosTest.tsx
it(scenarios.whenDelete, async () => {
    let item = tad.screenCapturing.getByTestId(testids.todoItem + "_0");
    await tad.userEventWaitable.click(tad.withinCapturing(item).getByTestId(testids.delete));
    
    await tad.userEventWaitable.click(tad.screenCapturing.getByTestId(testids.deleteNo));
    tad.cc("The Todo wasn't deleted");
    await tad.assertWaitable.equal(tad.withinCapturing(item).getByTestId(testids.todoItemCheckboxLabel).textContent, someTodos[0].text);
})

We have the result:

We added a new "perform action" line, simulating clicking on "No" (in the alert that's opened after clicking on "Delete"). And then a "verification" line. Here we have the same interception mechanism, but applied this time to assert from Mocha.

If the first lines were quite readable, this particular verification line is not straight forward. Meaning that just by reading it, it's not clear what we want to achieve actually. Hence the need for a comment, i.e. tad.cc() which is an alias for tad.currentComment(). We use comments this way (instead of // this way), so that they can be displayed in the UI.

Iterate: #4, #5, etc.​

To finish the test/scenario, we iterate between:

  • step 1: add data-testid (if needed)
  • step 2: copy/paste/adapt code for performing actions,
  • step 3: copy/paste/adapt code for verification.
tip

We recommend keeping "Auto run after page load" = true (default). Probably you are running TestsAreDemo inside a toolchain (such as Vite or Create React App) that has auto-refresh or hot module reload (HMR) enabled. Hence when you save, the browser will refresh and the test will run again. This is convenient.

Polish the demo for the end user​

The "slide show"/visual test steps are practically a demo of our component or app. They are meaningful for a dev that looks at the demo. But they are also (almost) meaningful for an end user. However, when presenting the demo to and user, a bit of additional polishing may be needed.

TodosTest.tsx
it(scenarios.whenDelete, async () => {
    tad.cc("A Todo may be deleted");
    
    tad.demoForEndUserHide();
    let item = tad.screenCapturing.getByTestId(testids.todoItem + "_0");
    await tad.userEventWaitable.click(tad.withinCapturing(item).getByTestId(testids.delete));
    
    await tad.userEventWaitable.click(tad.screenCapturing.getByTestId(testids.deleteNo));
    tad.cc("The Todo wasn't deleted");
    await tad.assertWaitable.equal(tad.withinCapturing(item).getByTestId(testids.todoItemCheckboxLabel).textContent, someTodos[0].text);
    tad.demoForEndUserShow();
    
    await tad.userEventWaitable.click(tad.withinCapturing(item).getByTestId(testids.delete));
    await tad.userEventWaitable.click(tad.screenCapturing.getByTestId(testids.deleteYes));
    tad.cc("The Todo was deleted. So the second element (index 1) became element the first element (index 0)");
    item = tad.screenCapturing.getByTestId(testids.todoItem + "_0");
    await tad.assertWaitable.include(tad.withinCapturing(item).getByTestId(testids.todoItemCheckboxLabel).textContent, someTodos[1].text); // include instead of equals, because we appended something in a step above
})

Some ways to polish:

  • by adding additional comments. Such as providing some context or intro at the beginning of the scenario, or adding text to mark the transition from the previous scenario.
  • by hiding steps which would bore the end user. E.g. here we don't want to show that we test clicking on the "No" button of the "Do you want to delete?" dialog.
  • by rephrasing a comment using a language suitable for the end user. E.g. here we used both "first element" (in "user language") and "index 0" (in dev language).

Iterate: scenario #2, #3, etc.​

We finished implementing and testing the first scenario/feature. And we also have a nice demo for it. It's time to do a bit of cleaning in TodosMeta. We remove the draft comments, because now this content is contained in the test, which is the single source of truth regarding how the feature works. We don't want to have comments in several places that we need to maintain (e.g. maybe in the future the alert will have a third orange button that does something new).
